We cannot use the assembler from gcc to compile the machine code files,
but I did already talk to the author of "yasm" and he seems to be interested in
creating a variant that is CLI compatible to /usr/ccs/bin/as and also 
inderstands the x86 syntax used by Sun.

Regarding GCC-4.x, the ON source compiles with GCC-3.4.3 but I'v heard that
the binaries no longer work. Somebody would need to check whether Schillix-ON
compiles with GCC-4.x and when this is done, somebody needs to check whether the
result works or not.


> Note that sunstudio-12.1 is buggy and does not create correct code for
> e.h. "hsfs" in 64 bit mode.

We now know what happens: SS12.1 does not properly auto-cast the computation 
from 32 bit ints to 64 bits ints that happen in the hsfs/RR time stamp 
conversion from the internal format to the UNIX format. It seems that Snorcle 
has a workaround using explicit casts.

The main reason for creating Schillix-ON is to have a ON base development 
project with the same goals as SchilliX. 

The goals for Schillix-ON are:

-	be close to the Solaris interfaces

-	to become fully OSS as soon as possible

-	to be independent from company interests

The SchilliX distro project is a different issue, the next things to do is to
compile Xorg and GNOME and to create packages. Packaging needs to be discussed
too, but this should be done separately.
